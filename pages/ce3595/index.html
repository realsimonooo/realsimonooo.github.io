# ⭐️ch1 基本概念

# 01 软件的定义、特点、分类

软件： 软件=程序+数据+文档

  • 程序： 事先设计的功能 性能要求执行 的 指令序列；
  • 数据： 进行通信、解释、处理而实用的信息表现形式；
  • 文档： 程序开发、维护、使用有关的图文材料；

# 软件的特点

  • 逻辑实体；抽象性；形态不可见；

    必须通过观察、分析、思考、判断来了解其功能、性能和其它特性；

  • 是人脑思维产物，生产过程与硬件不同；

    质量控制、软件产品保护问题；

  • 开发和运行受计算机控制；

    软件移植问题；

  • 开发技术落后，手工开发占统治地位；

    卡法效率低；

  • 软件工作涉及社会因素；

    机构、体制、管理方式；人的心理和观念；

软件的失效率和硬件不同：

软件需求与技术发展现状：

  • 复杂性越来越高，对软件人员的要求越来越高；
  • 复杂性与发展不适应的现象；

# 软件的分类

按软件结构与使用方式：

  • 独立式；
  • 组合式；
  • 集成式；
  • 嵌入式；

# 02 软件的发展和软件危机

# 软件的发展

  • 第一代：算法技术；
  • 第二代：结构化软件技术；
  • 第三代：面向对象技术；
  • 第四代：分布式面向对象技术；
  • 第五代：软件构件技术；

# 软件危机

开发和维护过程所遇到的一系列严重问题；

具体表现：

  • 开发进度难以预测；
  • 开发成本难以控制；
  • 用户对产品功能难以满足；
  • 产品质量无法保证；
  • 产品难以维护；

产生原因：

主观方面：开发方法不正确：

  • 用户需求不正确；
  • 缺乏正确的指导；

客观方面：逻辑部件规模庞大：

  • 开发规模越来越大；
  • 开发复杂度越来越高；

解决的途径：

  • 组织管理：
    • 项目管理方法；
    • 软件工程学；
    • 软件标准化；
  • 技术措施：
    • 开发技术与方法；
    • 开发工具；
    • 可重用性；

软件工程是从管理和技术两个方面研究如何更好地开发和维护计算机软件的学科；

软件危机推动软件工程发展；

# 03 软件工程方法学

# 什么是软件工程：

  • 工程： 将理论和所学知识应用于实践的科学；
  • 软件工程：应用计算机科学、数学、管理科学等原理，开发软件的工程；
  • 借鉴传统工程的原则、方法，以提高质量，降低成本为目的：
    • 计算机科学、数学用于构造模型与算法；
    • 工程科学用于指定规范、设计模型、评估成本；
    • 管理科学用于计划、资源、质量、成本等的管理；
  • 是一门交叉性的学科。

# 基本定义

指导计算机软件开发和维护的工程学科。采用工程的概念、原理、技术和方法来开发和维护软件，把经过时间考验而证明正确的管理技术和当前最好的技术方法结合起来。

# 基本原理

  • 使用分阶段的生命周期严格管理；
  • 坚持进行阶段评审；
  • 实行严格的产品控制；
  • 采用现代程序的设计技术；
  • 结果能清楚地审查；
  • 开发组人员小而精；
  • 承认不断改进实践的必要性

# 软件工程三要素

  • 方法：为软件开发提供如何做的技术。
  • 工具：提供支撑环境；
  • 过程：综合方法和工具进行软件开发；

# 基本目标

  • 较低开发成本；
  • 达标的软件功能；
  • 较好的软件性能；
  • 软件易于移植；
  • 较低的维护费用；
  • 按时完成开发，及时交付使用；

# 基本原则

  • 抽象；控制开发过程的复杂性
  • 信息隐蔽；
  • 模块化；
  • 局部化；
  • 确定性；
  • 一致性；
  • 完备性；
  • 可验证性；

# 过程特性

  • 可理解性；
  • 可见性；
  • 可支持性；
  • 可接受性；
  • 可靠性；
  • 健壮性；
  • 可维护性；
  • 速度；

# 过程及其内容

软件过程：为获得软件产品，在软件工具支持下由软件人员完成的一系列软件工程活动；

  • 规格说明；
  • 软件开发；
  • 软件确认；满足用户的要求；
  • 软件演进；用户变更要求；

# 开发方法的特点与选择

  • 结构化开发方法；
  • 面向对象方法；
  • 软件工程研究课题；

# ch2 软件生命周期及开发模型

# 软件的生命周期

一个软件从定义、开发、使用和维护，直到最终被废弃，所经历的生存过程；（软件的生存期）

# 生命周期划分

# 软件计划

  • 问题的定义：“要解决的问题”；
  • 可行性的研究：
    • 经济；
    • 技术；
    • 管理；
    • 社会环境；

# 需求分析

  • 确定系统逻辑模型；

  • 确定目标系统具备：

    • 功能；
    • 性能；
    • 接口；
    • 约束、限制条件；

# 软件设计

  • 总体设计方案：
    • 划分子系统功能；
    • 确定共享数据的组织；
  • 进行详细设计：
    • 处理模块设计；
    • 数据库系统设计；
    • 输入输出界面设计；
    • 编码设计；

# 软件测试

通过各种类型的测试，使达到预定的要求：

  • 单元测试；
  • 集成测试；
  • 验收测试；

# 软件维护

  • 日常运行管理；
  • 根据一定的规格进行必要的修改；
  • 评价系统的运行效率、工作质量、经济效益；
  • 运行费用和效果进行监理审计；

# 软件生存期模型

常见的开发模型：

  • 瀑布模型；
    • 包括： 计划、需求分析、设计、编码、测试、维护
  • 圆形模型；
  • 增量模型；
  • 构件组装模型；

# ⭐️ch3 需求分析与设计

# 可行性分析

  • 在当前环境、条件下，项目投资的研制工作是否具备必要的资源及其条件。
  • 可行性研究被广泛用于新产品开发、基建、工业、交通、商业设施等项目投资的各种领域；
  • 在大规模的系统开发之前，要从必要性和经济可行性、技术可行性、组织管理可行性、社会环境可行性等方面对未来系统的经济效益、社会效益进行初步的分析。

# ⭐️ 主要包括哪四个方面

  • 技术
  • 经济
  • 社会环境
  • 人员

# 任务与目的

  • 用最小的代价，在尽可能短的时间内确定问题是否能够被解决；
  • 确定问题是否能够解决和值得解决；
  • 分析可能的利弊关系；
  • 对行动的方针提出建议（是否可行）；

# 时间和成本

  • 实质上是在较高层次上以抽象方式进行系统分析和设计的过程；
  • 需要的时间长短取决于工程的规模；
  • 成本占软件工程的总成本的10%左右；

# 四个较为敏感的方面

  • 经济可行性：经济效益是否能超过开发成本；
  • 技术可行性：现有技术能否实现；技术风险的各种因素；
  • 操作可行性：用户的接受程度；
  • 法律可行性：是否合法，是否侵犯他人的利益；

# 分析的内容

  • 系统调研；
  • 政策与法律分析；
  • 资金与开发环境分析；
  • 技术分析；
  • 市场分析；
  • 风险分析；

# 可信性研究的步骤

  1. 系统地进行调研（复查系统的规模和目标）；

    调研的重点：

    • 了解用户与现行系统总的情况；
    • 现行系统与外部环境的联系；
    • 现行系统的现有资源；

  2. 现行系统分析；

    准备分析：

    • 确立分析计划；
    • 分析人员、任务分配；
    • 分析人员的必要培训；

    认识问题：

    • 了解系统应该解决的问题是什么；
    • 这些问题如何提出的？
    • 了解问题的结构；
    • 这些问题如何解决才能满足用户的要求；

    分析过程：

    1. 分析现实世界与理解当前系统，用具体的模型描述，获得当前系统的物理模型；
    2. 在理解“怎么做”的基础上，抽取“做什么”的本质，抽象出系统的逻辑模型；
    3. 分析目标系统与当前系统逻辑上的区别，导出目标系统的逻辑模型；
    4. 对目标系统逻辑模型进行补充，以求完整地描述。

    分析内容：

    • 环境分析；
    • 物理分析；
    • 功能分析；
    • 信息分析；
    • 动态分析；

  3. 建议新系统；

  4. 模型评审；

  5. 导出和评价可供选择的解决方案；

  6. 推荐方案并说明理由；

  7. 推荐行动方针；

  8. 书写文档提交审查

# 可行性分析的过程

  1. 定义问题；
  2. 分析问题；
  3. 导出解决方案；
  4. 评估、修改方案；
  5. 可行性报告；

# 需求分析

# 概念

从开发工作的流程看，是软件计划的下一步工作步骤，为软件设计阶段提供一个软件需求规格说明书。

# 为什么重要

  • 大型设备的失败，最终归结到需求分析；
  • 《用户需求规格书》是一个里程碑/极限；
  • 需求分析占软件开发工作量的30%左右；
  • 需求获取中的错误，会发散式地传播；

# ⭐️ 需求分析的重点

通过业务流程与数据流程的分析，需要在四个方面与客户达成完全一致目标；

  • 业务模型；
  • 功能模型；
  • 性能模型；
  • 接口模型；

需求分析要明确：万一需求有点变化，双方需要履行合同规定的需求变更管理程序；

# 需求分析的目标

软件规格说明：

  • 构建一个完整的、精致的目标逻辑模型；
  • 确定设计约束和定义软件有效性要求；

# 需求分析的任务

  • 认清问题、分析资料；
  • 建立分析模型；“逻辑模型”
  • 编写软件需求规格说明书：系统“做什么”

9个任务：

  1. 画出系统的组织结构图、列出各个部门的岗位角色（机构模型）；
  2. 画出系统业务的操作流程图；
  3. 列出系统的功能点，功能模型
  4. 列出系统的性能点，性能模型；
  5. 确定系统的运行环境高，环境模型；
  6. 确定系统的界面约定，界面模型；
  7. 对开发工期、费用、开发进度、系统风险等分析与评估。

# 需求分析的过程

  • 问题识别；
    1. 需求获取；
    2. 确定需求；
    3. 建立沟通管道
  • 分析与综合；
  • 编制需求分析阶段文档；
  • 需求分析评审；

# 获取需求的方法

需求获取困难，仍待解决：

  1. 用户需求具有动态性；
  2. 用户需求具有模糊性；
  3. 对需求达成一致的艰难性；
  4. 分析人员与用户的沟通问题；
  5. 管理体制、机构设置处在变革中；
  6. 分析方法与分析工具问题；

获取的常用方法

  1. 访谈/个别访问；
  2. 问卷调查/书面调查；
  3. 情景分析/电话/电视会议；
  4. 实地考察/收集资料
  5. 构造原型

# 需求分析的原则

  1. 解决逻辑问题；
  2. 以运行环境为基础；
  3. 用户参与的原则；
  4. 构造高质量的需求规格说明；

# 需求分析的基本要求

  1. 理解问题的数据域与功能域；
    • 数据域： 数据内容、数据结构、数据流；
      • 数据内容：数据项；
      • 数据结构：数据项的组织形式；
      • 数据流：数据通过系统时的变化方式；
      • 功能：对数据进行的一系列的转换；
      • 接口：两个功能之间的数据传递；
  2. 自顶向下、逐层分解；
    • 分解：将复杂问题分解为若干相对简单的问题；
    • 目的：降低问题求解的复杂性；
    • 便于理解和分工完成，再装配起来形成一个完整的系统。
    • 终点：可以使软件开发走向构件式的开发；
  3. 给出系统的逻辑视图和物理视图；
    1. 逻辑视图：描述软件要达到的功能和要处理的信息之间的关系，但没有描述实现的细节；
    2. 物理视图：给出的是处理功能和信息结构的实际表现形式，需要考虑实际的环境和具体的设备；

# 需求分析的方法

  • 面向数据流的分析方法；
  • 面向功能的分析方法；
  • 面向数据的Jackson方法；
  • 面向对象的分析方法；

# 需求分析方法的共性

  • 支持数据域分析机制
    • 数据流、数据字典、数据结构；
  • 功能表示方法
    • 功能用加工表示、语言描述
  • 接口的定义
  • 问题的分解及抽象化
  • 逻辑视图、物理视图
  • 系统抽象模型

# 结构化需求分析方法

分析模型：

  • 数据模型；
  • 功能模型；
  • 行为模型；

两种主要的建立分析模型的方法：

  • 结构化分析模型；
  • 面向对象分析模型；

结构化分析建模方法：

  • 分析模型：

    从DD出发，通过ER图（实体关联图）、DFD图（数据流）、STD图（状态转换图）构建数据模型、功能模型；

  • ⭐️数据流图：【核心】

    表达系统内部数据的运动情况；

  • 数据字典：

    定义系统的数据；

  • 描述数据加工：

    • 结构化英语
    • 判定表和判定数

  • 实体关联图：构建数据模型

  • 特点：利用数据流图来帮助理解问题、对问题分析；利用图形工具来模拟数据处理的过程。

# 数据流图
  • 从数据传递和加工的角度，以图形化的方式刻画数据流从输入到输出的移动和变换过程；
  • 数据流图中具体的物理元素都已经去掉，只剩下存储、流动、加工和使用情况；
  • 图形表示逻辑系统。
# ⭐️设计数据流图的基本原则

  1. 所有图形必须是四种元素之一：

    数据流、加工处理、文件、数据源或终点；

  2. 主图必须含有四种元素，缺一不可；

  3. 主图必须分布在外部实体之间，实体可以有多个；

  4. 加工框至少有一个输入数据流和一个输出数据流；

  5. 不考虑初始化，每一个元素都必须有名字；

  6. 任何一个数据流子图必须与父图上的一个加工框对应，两者的输入/输出流必须一致

  7. 可以在数据流图中加入物质流；

  8. 不可在数据流图中夹带控制流；

  9. 初画时可以忽略一些细节，基本原则先外后内。

# 数据字典

数据流图的基础上，每个数据流、文件、数据项加以定义所组成的集合称为数据字典（Data Dictionary）

编写的基本要求：

  • 数据流图的基础上的各种元素的定义必须明确，一致、易于理解；
  • 命名、编号应该与数据流图一致；
  • 数据流图的成分定义与说明没有遗漏，无同名异义或者异名同义。
  • 格式规范、文字精炼、符号正确。

6种条目形式：

  • 数据项：数据处理中的基本的不可分割的逻辑单位；
  • 数据结构：描述数据流和数据存储的逻辑组成；
  • 数据流：定义数据流图中的数据流，包括名字、编号、来源去向、包含的数据结构名称等；
  • 数据存储：文件的名称、编号、组成、记录数；
  • 处理功能：说明加工的名称、编号，I/O数据流，加工逻辑概括描述等。
  • 外部实体：说明外部实体的名称、编号，I/O数据流等；

# 判定表
  • 采用表格形式表达逻辑判断问题；
  • 党数据流图处理需要依赖多个逻辑条件的取值，这些取值可能构成多种不同的情况，相应执行不同的动作；这些问题适合使用判定树、判定表作为描述加工小说明的工具；
  • 为说明复杂的决策逻辑提供了一种描述机制，以一种形式化的表格为基础；

构造步骤：

  • 计算可能的所有条件组合，通常n个条件最多可以有个条件组合书；
  • 列出所有基本条件和基本动作；
  • 对每一种状态找出所有的条件，填入条件项；
  • 对每一种规则制定动作，填入动作项；
# 判定树
  • 是判定表的一种变化，呈属性的图形工具，适合于描述处理具有多种测虐，要根据若干条件的判定，确定所采用策略的情况；
  • 优点：清晰、直观。
  • 但当条件多、互相组合时，不容易清楚地表达判断过程；

# 软件结构与过程

软件过程：集中考虑每个模块各自的处理细节；

​ 过程必须提供一个精确的处理规格说明，包括事件顺序，精确的判定位置，重复的操作，数据组织和结构。

结构和过程之间的关系：描述一个模块的实现，必须包括所有从属于该模块的所有模块的饮用，从属于该模块的所有模块是通过结构体现的。

软件结构与过程：用数据、功能和行为模型表示的软件需求，采用某种设计方法进行数据设计、体系结构设计、接口设计、过程设计；

# 软件设计的重要性

软件设计是开发阶段最重要的步骤，是软件开发过程中质量得以保证的关键步骤；

  • 数据设计：将E-R（实体关系图）中描述的对象和关系以及数据词典中描述的详细数据内容转化为数据结构的定义；
  • 体系结构设计：定义软件系统各主要成分之间的关系；
  • 接口设计：根据数据流图定义软件内部各成分之间的关系；
  • 过程设计：把结构成分转换为软件的过程性描述；

# 软件设计过程的目标

  • 必须实现分析模型中描述的所有显式需求，必须满足用户希望的所有隐式需求；
  • 必须是可读的、可理解的，将来易于编程、易于测试、易于维护；
  • 应从实现角度出发，给出与数据、功能、行为相关的软件全貌。

# 衡量的技术标准

  • 设计出来的结构应该是分层结构，建立软件成分之间的控制；
  • 设计应当模块化，从逻辑上讲软件划分为完成特定功能或子功能的构件；
  • 应当既包含数据抽象，也包含过程抽象；
  • 应当建立具有独立功能特征的模块；
  • 应当建立能够降低与外部环境之间复杂连接的接口；
  • 应能根据软件需求分析获取的信息，建立可驱动可重复的方法。

# 树状结构和网状结构

树状的特点：只有一个顶层模块；同一层模块之间不发生联系；

网状的特点：任意两个模块间都可以有双向的关系；分不出来层次；

# 模块化
  • 指整个系统被划分成若干个单独命名和可编址的部分，称之为模块。模块可被组装起来满足整个问题的需求；
  • 把问题/自问题的分解与软件开发中的系统/子系统或系统/模块对应起来，就能把大而复杂的软件划分成易于理解的模块结构；

⭐️模块的基本属性

四个基本属性：

  • 功能：描述模块做什么；
  • 逻辑：模块内部怎么做；
  • 状态：模块使用时的环境和条件；
  • 接口：模块的输入与输出；

模块的特征：

  • 抽象：层次方式构造和分析复杂系统；
  • 逐步求精；
  • 信息隐蔽；
  • 局部化；（关系密切）
# 模块的独立性
  • 每个模块只涉及软件要求的具体的子功能，和其他的模块的接口是简单的；
  • 一个模块只具有单一的功能且与其它模块没有太多的联系，则称此模块具有模块独立性；
  • 模块独立性是软件质量的关键。

模块独立性的度量

由模块间的耦合与内聚度量；

  • 耦合：各模块的互相连接的紧密程度。
  • 内聚：各功能元素彼此结合的紧密程度；

要尽量做到高内聚，低耦合。

对于大规模的复杂软件系统，系统结构设计和规格说明和算法与数据结构相比，结构设计变得越来越重要。

# 定义

  • 为软件系统提供了一个结构、行为和属性的高级抽象。由构成系统的元素的描述、这些元素的相互作用、指导元素集成的模式以及这些模式的约束组成；
  • 不仅指定了系统的组织结构和拓扑结构，并且显示了系统需求和构成系统的元素之间的相应关系，提供了设计决策的基本原理。

# 组成

是一种可预制和重构的软件框架结构：

​ 体系结构 = 构件 + 连接件 + 约束

  • 构件：可预制和重构的软件部分，是组成体系结构的基本单元或数据存储单元；
  • 连接件：可预制和重构的软件部分，是构件之间的连接单元；
  • 约束：描述连接和连接件之间的关系。

# 结构的划分

按照水平方向或垂直方向：

  • 水平方向：
    • 按主要的程序功能来定义模块的各个分支；
    • 顶层模块是控制模块，用来协调程序各个功能之间的通信和运行；
    • 下级模块的最简单的水平划分方法是建立三个分支；
      • 输入、处理（数据变换）、输出；
  • 垂直方向：
    • 工作自顶向下逐层分布：
    • 顶层模块执行控制功能；
    • 底层模块是实际输入、计算和输出的具体执行者；
    • 方便维护。

# 三要素

  • 程序构建的层次结构；
  • 构件之间交互的方式；
  • 数据的结构；

# 应保持的性质

  • 结构：定义系统的构建，以及这些构件打包的方式和相互交互的方式；
  • 附属的功能：如何实现对功能、性能、可靠性、安全性、适用性以及其它系统需求；
  • 可复用：可复用的模式。以便于在以后类似的系统族的设计中使用它们。

# 对象和类

对象的概念的面向对象技术的核心。对象是现实世界中一个世纪存在的事物，是构成客观世界的一个独立单位，可以是有形的，也可以是无形的。

软工中，主要专门指它们在系统中的抽象表示。

# 对象的定义

是客观事物或概念的抽象表述，对象是系统中用来描述客观事物的一个实体，是构成系统的一个基本单位。

一个对象由一组属性和对这组属性进行操作的一组服务构成。属性和服务是构成对象的主要因素。

  • 对象的属性：描述对象静态特征的数据项。是对象本身的性质，也可以是另一个对象；
  • 对象的操作：描述对象动态特征的操作序列。规定了对象的行为，对象搜能提供的服务；

# 类的定义

是一组具有相同属性和相同操作的对象的集合。类是抽象的，对象是具体的。所以又称，对象是类的一个实例。

  • 类的属性：是对象的状态的抽象，用数据结构来描述类的属性；
  • 类的操作：是对象的行为的抽象，操作实现的过程称为方法，方法有方法名，方法体和参数。

# 面向对象的基本特征

一个重要的思想：尽可能模拟人类习惯的思维方式来进行软件开发，使得描述的问题域和实现解法的解空间在结构上一致。基本特征：抽象、封装、继承和多态；

# 抽象

强调实体的本质、内在的属性；在系统开发中，抽象指的是再决定如何实现对象之前的对象的意义和行为。使用抽象可以尽可能避免过早考虑一些细节。

类实现了对象的数据（状态）和行为的抽象；

# 封装

是一种信息隐蔽技术，利用抽象数据类型将数据和基于数据的操作封装在一起；

用户看得见对象封装界面上的信息，对象的内部的实现对用户是隐蔽的。封装的目的是使对象的使用者和生产者分离，使对象的定义和实现分开。

# 继承

是使用现存的定义作为基础，建立新定义的技术；

继承是类与类的基本关系，基于层次关系的父类和子类之间共享数据和操作的一种机制。父类中定义了其所有子类的公共属性和操作，在子类中除了定义自己特有的属性和操作，可以继承父类的属性和操作，还可以对父类中的操作重新定义其实现的方法。

# 多态性和动态绑定

多态是多种形态，指同一个操作用于不同对象上可以有不同的解释，并且产生不同的执行结果。多态性实际上提供了一种具体情况具体分析的问题解决方案。

动态绑定是运行时，根据对象接收的信息动态地确定要连接的服务代码。

# 消息与继承

消息：向对象发出的服务请求（互相联系、协同工作等）；

继承：使用现存的定义作为基础，建立新定义的技术。分类如下：

  • 单重继承：一个子类只有一个父类；
  • 多重继承：一个子类可以有多父类，继承多个父类的数据结构和方法；

# 抽象类和永久对象

  • 抽象类：没有实例的类，把一些类组织起来，提供一些公共的行为，但是并不需要使用这个类的实例，而仅仅使用其子类的实例；
  • 永久对象：指生存期可以超越程序的执行时间而长期存在的对象；

# 面向对象的基本定义

  • 是一种使用对象、消息传递、类、继承、多态和动态绑定来开发问题域模型的范型；
  • 一种基于对象、类、实例和继承等概念的技术；
  • 用对象作为建模的原子；
  • 用来描述一种把软件组织成对象集合的软件开发策略，对象中既包括数据也包括操作。

# ⭐️面向对象方法的优点

与传统的软件工程方法相比：

  1. 与人们习惯的思维方式一致；
  2. 降低系统的复杂度；
  3. 提高软件系统的可重用性；
  4. 提高软件的可维护性和可扩充性；
  5. 隔离变化；

  • “一般—具体结构”：分类结构，“或”关系
  • “整体—部分结构”：组装结构，“与”关系

# 模型

显示客观世界的形状或者形状的抽象模拟和简化，是系统的一个抽象，提供了系统的骨架和蓝图。

为人们提供了系统的各个部分是如何阻止其俩的，模型可以是抽象的或想醒的。一个好的模型注重与某种特定情形相关的方面，而忽略其它细节，每个系统的不同方面利用不同的模型来描述，结构方面和动态方面。

# 建模

为正在开发的系统制定一个精确、简明和易于理解的对象模型。为了更好地理解正在开发的系统，达到以下的目的：

  • 模型有助于按照实际情况或按照所需要的样式对系统进行可视化；
  • 模型能够制约系统的结构和行为；
  • 模型给出了指导构造系统的模版；
  • 模型对做出的决策进行文档化；
